# 지속거래약화 Y 전환 설계

> 상태: 사용자 승인
> 기준일: 2026-07-13
> 기준 문서: `y_setting_pipeline.md`

## 1. 목표

프로젝트의 최종 Y를 기존 `Y_핵심관계약화_3개월`에서 `Y_지속거래약화_3M70`으로 완전히 교체한다. 새 Y는 실제 해지나 법적 의미의 이탈이 아니라, 36개월 완전관측 법인의 월별 거래활동에서 확인되는 지속거래약화 proxy다.

## 2. 분석 대상

- 기간: 2023-01~2025-12
- 대상: 36개월이 모두 관측된 법인
- 현재 기준 법인 수: 3,372개
- 누락 고객-월을 0으로 보충하지 않는다.
- 법인별 월 중복, 월 연속성, 정확히 36개월 관측 여부를 라벨 계산 전에 검증한다.

## 3. 핵심거래활동금액

법인별·월별 핵심거래활동금액은 다음 원천 변수의 합으로 계산한다.

```text
입출금활동금액 = 요구불입금금액 + 요구불출금금액

채널활동금액 = 창구거래금액
                + 인터넷뱅킹거래금액
                + 스마트뱅킹거래금액
                + 폰뱅킹거래금액
                + ATM거래금액

카드활동금액 = 신용카드사용금액 + 체크카드사용금액

핵심거래활동금액 = 입출금활동금액
                    + 채널활동금액
                    + 카드활동금액
```

외환, 수신, 여신, 자동이체, 상품관계폭은 최종 Y 판정식에 포함하지 않는다. 필요하면 향후 모델 feature 또는 설명 변수로만 사용한다.

원천 금액의 결측은 관측된 0과 구분한다. 구성 변수 중 하나라도 결측이면 해당 월 핵심거래활동금액과 그 값에 의존하는 판정을 결측으로 둔다.

## 4. 월별 YoY와 drop50

법인별 월 순서를 기준으로 12개월 전 핵심거래활동금액과 비교한다.

```text
핵심거래_YoY_ratio(t) = 핵심거래활동금액(t) / 핵심거래활동금액(t-12)
drop50(t) = 핵심거래_YoY_ratio(t) < 0.50
```

분모가 0이거나 분자·분모가 결측이면 YoY ratio와 drop50을 판정 불가능으로 둔다. `ratio == 0.50`은 drop50이 아니다.

## 5. 3개월 연속 이벤트

`drop50=1`이 3개월 연속인 구간에서 세 번째 달을 이벤트 충족월로 정의한다. 같은 연속 구간의 네 번째 이후 달을 새 이벤트로 반복 계산하지 않는다.

```text
core_3m_event(t) = drop50(t-2) = drop50(t-1) = drop50(t) = 1
                   AND 직전 월에 3개월 연속 상태가 완성되어 있지 않음
이벤트월 = t
```

감소가 시작된 첫 달이 아니라 3개월 연속 조건이 처음 완성된 달이 이벤트월이다. 회복 후 다시 별도의 3개월 연속 구간이 만들어지면 새 이벤트 후보로 인정한다.

## 6. 이벤트 전후 지속성 검증

각 이벤트 후보에 다음 값을 계산한다.

```text
baseline12 = 이벤트월 직전 12개 월(t-12~t-1)의 핵심거래활동금액 평균
future3 = 이벤트월 직후 3개 월(t+1~t+3)의 핵심거래활동금액 평균
future3_to_baseline = future3 / baseline12
```

두 창은 이벤트월을 포함하지 않는다. 각 창의 모든 월이 존재하고 핵심거래활동금액이 판정 가능해야 한다. `baseline12 <= 0`이거나 미래 3개월이 부족하면 지속성 조건과 최종 Y를 판정 불가능으로 둔다.

## 7. 최종 Y 계약

```text
Y_지속거래약화_3M70 = 1
if
핵심거래_YoY_ratio < 0.50이 3개월 연속 발생
AND
future3_to_baseline < 0.70
```

이벤트 후보가 있으나 `future3_to_baseline >= 0.70`이면 해당 이벤트의 Y는 0이다. `future3_to_baseline == 0.70`도 Y=0이다. 필수 관찰창이나 분모가 부족하면 Y는 0이 아니라 결측이다.

라벨 구현은 두 결과를 제공한다.

1. 법인×월 이벤트 패널: 모든 관측월과 이벤트 후보·지속성 검증 결과
2. 이벤트 테이블: 3개월 연속 조건이 처음 충족된 이벤트 후보만 포함

주요 산출 컬럼은 다음과 같다.

```text
법인ID
기준년월
입출금활동금액
채널활동금액
카드활동금액
핵심거래활동금액
핵심거래_YoY_ratio
drop50
drop50_연속개월수
core_3m_event
이벤트이전12개월평균
이벤트이후3개월평균
future3_to_baseline
Y_지속거래약화_3M70
지속거래약화사건ID
```

사건 ID는 `법인ID+이벤트월`로 구성한다.

다음 활성 자료를 새 정의로 통일한다.

- `financial_dormancy.md`
- `README.md`
- `AGENTS.md`

기존 5개 핵심축·3축 동시감소 기반 `Y_핵심관계약화_3개월`은 최종 Y와 현재 성능으로 표현하지 않는다. 필요한 경우 과거 후보 실험으로만 명시한다. 해당 Y로 계산한 PR-AUC, 사건 Recall 등의 성능 수치는 새 Y 모델을 재학습하기 전까지 현재 프로젝트 성과에서 제거한다.

## 9. 구현 구조

라벨 계산은 모델 코드와 분리한다.

```text
src/preprocessing/persistent_transaction_weakening_labels.py
    - 설정과 필수 컬럼 계약
    - 코호트·월 연속성 검증
    - 핵심거래활동금액 계산
    - YoY, 연속 이벤트, 전후 창, 최종 Y 계산

src/preprocessing/run_persistent_transaction_weakening_labels.py
    - 입력 로드
    - 라벨 함수 호출
    - 패널·이벤트 결과 저장

tests/test_persistent_transaction_weakening_labels.py
    - 경계값과 결측·0 분모·반복 이벤트·관찰창 검증
```

모델링 코드는 라벨 구현과 문서 전환이 끝난 뒤 새 라벨 계약에 맞춰 수정한다. 새 모델 성능은 실제 데이터로 재학습·시간 검증한 결과만 기록한다.

## 10. 오류 처리

- 필수 컬럼 누락: 누락 컬럼명을 포함한 `ValueError`
- 법인×월 중복: 중복 키 예시를 포함한 `ValueError`
- 법인별 36개월 미충족 또는 월 불연속: 해당 법인 수를 포함한 `ValueError`
- 음수 거래금액: 원천 데이터 오류로 간주하고 `ValueError`
- 날짜 파싱 실패: 실패한 원본 값 예시를 포함한 `ValueError`

## 11. 테스트 및 완료 기준

- 합산 금액이 정확하다.
- 12개월 전 값과만 YoY를 계산한다.
- `0.49`는 drop50, `0.50`은 비-drop50이다.
- 정확히 세 번째 연속 월만 이벤트가 된다.
- 이벤트 이전 12개월과 이후 3개월에서 이벤트월이 제외된다.
- `0.69`는 Y=1, `0.70`은 Y=0이다.
- 분모 0, 결측, 미래창 부족은 Y 결측이다.
- 회복 없이 이어지는 네 번째 감소월은 중복 이벤트가 아니다.
- 회복 후 새 3개월 연속 구간은 별도 사건이다.
- 전체 테스트가 통과하고 활성 문서에서 기존 Y를 현재 최종 Y로 설명하는 문구가 남지 않는다.

## 12. 범위 경계

이번 구현은 최종 이벤트 라벨과 관련 문서 전환까지 수행한다. 기준월 `t`에서 미래 이벤트를 예측하는 별도 rolling 조기예측 target, 모델 재학습, CRM 우선순위 재산출은 이 라벨 구현 이후 단계다. 사후 확인에 미래 3개월을 사용하는 값을 기준월 feature로 사용하지 않는다.
