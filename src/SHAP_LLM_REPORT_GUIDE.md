# SHAP 기반 RM 전략 보고서 운영 가이드

## 1. 목적과 책임 경계

이 서비스는 향후 6개월 `Y_INTERVENE_M12_v2` 지속거래약화 가능성을 바탕으로 RM의 조기관리와
접촉 전략을 돕는다. 각 구성요소의 책임은 아래와 같이 분리한다.

- LightGBM과 Isotonic 보정: 위험점수를 계산한다.
- 저장 Local SHAP: 해당 고객의 모델 예측 기여도를 설명한다.
- 규칙 엔진: 저장 SHAP 행을 검증하고 변수의 그룹·방향·의미를 결정한다.
- Gemini: 검증된 정량 근거를 이용해 6개 고정 섹션의 한국어 초안을 작성한다.
- API와 PDF: 저장 근거와 보고서의 정량 항목을 대조하고, 화면과 PDF에 결과를 표시한다.

웹 요청 중 모델, CLV, SHAP을 다시 계산하지 않는다. Gemini는 위험확률, CLV, SHAP 또는
변화율을 재계산하지 않으며, 컨텍스트에 없는 사실을 만들지 않는다. AI 초안의 최종 판단과 고객
접촉 결정은 RM에게 있다.

## 2. 운영 모델 계약

운영 모델의 피처 세트는 `FS2_R1_DACK_DYNAMIC`이며 **총 56개**다. LightGBM이 이 피처만
사용하고, 위험확률은 Isotonic 보정 후 제공한다. 구성은 **16개 기본 피처 + 40개 동적 피처**다.

| 블록 | 개수 | 내용 |
|---|---:|---|
| 관계·수준·구성 기본 피처 | 16 | 핵심거래, 수신·여신 관계, 상품·채널 활성 폭, 구성 균형, 좌수·카드·자동이체 강도 |
| D/A/C/K 동적 피처 | 40 | 네 개 축 각각의 최근 비교, 추세, 변동성, 활동 빈도, 감소·비활성 신호 |

축의 의미는 다음과 같다.

- 요구불: 요구불입금금액과 요구불출금금액 활동
- 자동이체: 자동이체금액 활동
- 채널: 창구·인터넷뱅킹·스마트뱅킹·폰뱅킹·ATM 거래 활동
- 카드: 신용카드사용금액과 체크카드사용금액 활동

업종·지역·고객등급·전담여부는 모델 입력이 아니다. 이 값들은 보고서의 고객 맥락과 추천 선택에는
사용될 수 있지만, 위험점수의 직접 원인 또는 SHAP 피처로 설명해서는 안 된다. 세그먼트도 모델
입력이 아니며 설명과 추천을 위한 맥락이다.

## 3. SHAP 해석의 고정 원칙

- `impact > 0`은 이 고객의 모델 위험점수를 높이는 방향(`risk_up`)이다.
- `impact < 0`은 모델 위험점수를 낮추는 방향(`risk_down`)이다.
- `impact == 0`은 방향을 단정하지 않는 중립(`neutral`)이다.
- SHAP은 보정 전 모델 예측 기여도이며 인과관계나 확률 변화량이 아니다.

따라서 `SHAP 0.31이 위험확률을 31% 높였다`, `요구불 감소 때문에 이탈한다`와 같은 표현은 금지한다.
허용되는 표현은 “요구불 활동 관련 신호가 모델 위험점수를 높이는 방향으로 기여했습니다”처럼
모델 기여도와 방향을 분리한 표현이다. `risk`는 실제 해지·부도·확정 휴면 확률이 아니라 향후
6개월 지속거래약화 발생 가능성이다.

### 3.1 비중의 의미

`top10AbsShare`는 다음과 같이 **선택된 Top 10 안에서만** 계산한다.

```text
top10AbsShare = abs(해당 SHAP) / sum(abs(Top 10 SHAP))
```

그러므로 이는 전체 모델 피처의 기여 비중이나 확률 변화량이 아니다. 그룹의
`top10AbsShare`도 같은 Top 10 분모에서의 절대 SHAP 비중이다. 그룹의 `signedShap`은 방향을,
`absoluteShap`은 영향 강도를 보조적으로 나타낸다. 한 그룹에 양·음 기여가 함께 있으면
`mixed`로 표시해 signed 합계의 상쇄만으로 해석하지 않는다.

## 4. 피처 그룹과 번역 사전

기본 피처는 아래 그룹으로 해석한다.

| 그룹 | 기본 피처 | RM 표현 |
|---|---|---|
| 관계 폭·구성 | `핵심거래_수준`, `관계영역_활성폭`, `핵심거래_입출금대채널_평균비중` | 핵심 거래관계의 수준·폭·구성 |
| 수신 관계 | `수신자산_수준`, `수신상품_활성폭`, `수신_요구불대기타_로그균형`, `수신좌수_강도` | 수신 관계의 수준·상품 폭·구성 |
| 여신 관계 | `여신관계_수준`, `여신세부상품_활성폭`, `여신_운전대시설_로그균형`, `여신부모좌수_강도` | 여신 관계의 수준·상품 폭·구성 |
| 채널 이용 | `채널_활성폭`, `채널_비대면대창구_로그균형` | 채널 이용의 폭·구성 |
| 자동이체 | `자동이체건수_강도` | 자동이체 이용 강도 |
| 카드 관계 | `신용카드개수_강도`, `카드_활성률` | 법인카드 관계와 활동월 빈도 |

동적 피처는 `요구불`, `자동이체`, `채널`, `카드` 축에 다음 정확한 접미사를 붙인다. 각각의 축은
차례대로 입출금 활동, 자동이체, 채널 이용, 카드 관계 그룹에 속한다.

| 접미사 | 규칙 엔진의 한국어 의미 | RM 문장 기준 |
|---|---|---|
| `최근3대직전3_로그차이` | 최근 3개월과 직전 3개월 비교 | 최근 3개월과 직전 3개월의 활동 비교 |
| `최근3대이전9_로그차이` | 최근 3개월과 이전 9개월 비교 | 최근 3개월과 이전 9개월의 활동 비교 |
| `H2대H1_로그차이` | 최근 반기와 앞선 반기 비교 | 최근 반기와 앞선 반기의 활동 비교 |
| `로그변동성` | 최근 12개월 활동 변동성 | 최근 12개월 활동의 변동성 |
| `TheilSen_추세` | 최근 12개월 강건 추세 | 최근 12개월 전반 추세 |
| `최근3대이전9_활성률차이` | 활동월 빈도 변화 | 거래가 발생한 월의 빈도 변화 |
| `월간50퍼감소횟수` | 최근 12개월 월간 50% 이상 감소 횟수 | 월간 급감 반복 신호 |
| `월간감소율` | 전월보다 감소한 월의 비율 | 감소한 월의 비중 |
| `최대연속비활성개월` | 최장 연속 비활성 기간 | 연속 비활성 기간 |
| `현재월대직전6_로그차이` | 현재월과 직전 6개월 평균 비교 | 현재월과 직전 6개월 평균의 비교 |

이 사전은 변수명 설명을 위한 것이며, SHAP 부호와 무관하게 “값이 낮으면 항상 위험”이라고
일반화하지 않는다. 해당 고객의 저장 `impact` 방향을 우선한다.

## 5. Local SHAP Top 10 전처리와 보존 규칙

입력은 고객별로 저장된 **Local SHAP Top 10**이며 1~10개만 허용한다. 운영 시 일반적으로 10개가
저장되지만, 유효한 입력은 적어도 1개여야 한다. 규칙 엔진은 다음을 검사한다.

1. 모든 `feature`가 56개 `FS2_R1_DACK_DYNAMIC` 피처에 속하는지 검사한다.
2. 모든 `impact`가 유한한 수인지 검사한다.
3. `rank`가 1부터 중복·누락 없이 연속하는지 검사한다.
4. `rank` 오름차순으로 정렬하고, 각 행에 `group`, `meaning`, `direction`, `top10AbsShare`를 추가한다.
5. 같은 그룹의 `signedShap`, `absoluteShap`, `top10AbsShare`, 포함 순위와 절대값 상위 두 대표 피처를 계산한다.

**Top 10 원본** `shapFactors`는 authoritative context에서 변경하지 않는다. 반면 파생
`localShapTop10`은 검증된 `rank` 오름차순으로 정렬한다. 이 파생 행은 `feature`, `featureValue`,
`impact`, `rank`를 포함한 모든 원본 필드를 보존하고 `group`, `meaning`, `direction`,
`top10AbsShare`만 추가한다. 그룹화는 LLM 서술 중복을 줄이기 위한 추가 맥락일 뿐 원본을 대체하지
않는다. UI와 PDF는 저장된 원본 행을 모두 표시하며 운영 적격 고객은 통상 10개를 표시한다.

`featureValue`는 nullable이다. 값이 `null`이면 모델 입력값, 금액, 건수, 실제 변화율을 알 수 없다는
뜻이다. 프롬프트는 값·금액·변화율을 추정하지 말라고 지시한다. 다만 현재 후처리만으로 생성된 모든
숫자가 근거에 기반했는지 완전히 검증하지 못한다. 따라서 `featureValue`가 `null`인 경우에는 피처명·의미·
SHAP 방향만을 근거로 삼고, 생성 문장의 수치와 사실은 RM 또는 원천 근거 확인이 필요하다. UI와 PDF는
`null` 값을 `-`로 표시한다.

## 6. Gemini 프롬프트 입력 계약

`build_strategy_report_prompt`는 저장된 보고서 컨텍스트와 규칙 엔진이 만든 `shapAnalysis`를 Gemini에
전달한다. 아래는 형식 예시이며, 숫자는 예시일 뿐 계산을 Gemini에 맡기지 않는다.

```json
{
  "customer": {
    "id": "CORP-AG534",
    "name": "알파코",
    "risk": 82.0,
    "clvRisk": 123456.78,
    "potentialLoss": 45678.9,
    "signals": []
  },
  "recommendation": {
    "contact": "RM 직접 접촉",
    "action": "거래 감소 원인 확인",
    "reason": "복합 약화 신호"
  },
  "strategySummary": "저장된 추천 요약입니다.",
  "shapFactors": [
    {
      "feature": "요구불_TheilSen_추세",
      "featureValue": null,
      "impact": 0.42,
      "rank": 1
    }
  ],
  "shapAnalysis": {
    "featureSet": "FS2_R1_DACK_DYNAMIC",
    "featureCount": 56,
    "localShapTop10": [
      {
        "feature": "요구불_TheilSen_추세",
        "featureValue": null,
        "impact": 0.42,
        "rank": 1,
        "group": "입출금 활동",
        "direction": "risk_up",
        "meaning": "최근 12개월 강건 추세",
        "top10AbsShare": 1.0
      }
    ],
    "groupedSignals": [
      {
        "group": "입출금 활동",
        "signedShap": 0.42,
        "absoluteShap": 0.42,
        "top10AbsShare": 1.0,
        "direction": "risk_up",
        "includedRanks": [1],
        "representativeFeatures": ["요구불_TheilSen_추세"]
      }
    ]
  }
}
```

Gemini는 모든 원시 Top 10 행을 권위 있는 근거로 검토하되, 같은 그룹의 유사 피처는 종합 신호로
묶어 반복 문장을 피한다. 프롬프트는 컨텍스트에 없는 값, 금액, 변화율, 상품 보유, 사건, 재무정보를
생성하지 말라고 지시한다. 이 지침만으로 모든 생성 수치의 근거성이 증명되는 것은 아니므로 RM은 원천
근거를 확인한다. 업종·지역·고객등급·전담여부를 모델 피처로 부르지 않으며, 추천은 확정 처방이 아닌
RM의 조기 확인·접촉 전략으로 작성한다.

## 7. Gemini 출력과 후처리

Gemini 출력은 다음 6개 필드를 가진 JSON이어야 한다.

```json
{
  "riskSummary": "조기관리 필요성을 요약한 문장",
  "valueAssessment": "CLV_Risk와 PotentialLoss의 시나리오 해석",
  "weakeningDrivers": "종합한 주요 약화·보호 신호",
  "contactStrategy": "RM 확인 중심의 접촉 전략",
  "recommendedActions": ["우선 실행할 RM 확인사항"],
  "caveats": ["모델 및 가치 추정의 한계"]
}
```

후처리는 제한적이고 결정론적이며, 프롬프트 지침 전체를 포괄적으로 집행하는 필터가 아니다.

- 개별 해지 확률, 부도 확률, 휴면 확률 표현은 지속거래약화 가능성으로 정규화한다.
- 특정 피처의 방향과 반대되는 위험·보호 설명은 해당 피처 이름이 있는 local clause에서 거부한다.
- SHAP과 정량 %, %p, bp 또는 숫자-만큼 표현을 같은 local clause에서 확률·위험 변화 또는 인과로
  연결한 문장은 거부한다.
- 중립 피처에 위험 상승 또는 보호 방향을 단정하는 문장은 거부한다.
- 사용자 한계문구 중 중복되는 내용을 제거한 뒤 아래 표준 한계문구를 항상 덧붙인다.

일반적인 인과 표현은 프롬프트에서 금지하지만 후처리가 빠짐없이 탐지하지는 않는다. 예를 들어
피처명이나 SHAP 수치와 연결되지 않은 원인 단정, 또는 위 정량 패턴 밖의 표현은 현재 필터만으로
전부 차단된다고 보장할 수 없다. 그러므로 생성 결과의 근거·인과 표현은 RM과 원천 데이터가 검토한다.

표준 한계문구는 다음 세 문장이다.

1. `Y는 실제 해지·부도·확정 휴면이 아닌 지속거래약화 proxy 예측입니다.`
2. `SHAP은 인과관계나 확률 변화량이 아닌 모델 예측 기여도입니다.`
3. `CLV_Risk와 PotentialLoss는 확정 손실액이 아닌 시나리오 추정치입니다.`

## 8. API·화면·PDF 계약

`POST /api/reports/{corporate_id}/generate`는 적격 고객의 저장 위험, `CLV_Risk`, `PotentialLoss`,
추천 및 저장 SHAP을 근거로 Gemini 초안을 만들고, API가 동일 컨텍스트의 정량 항목과 SHAP 원본을
응답에 다시 결합한다. 보고서는 DB에 저장하지 않는다.

`POST /api/reports/{corporate_id}/pdf`는 화면에서 전달된 보고서의 법인ID, 고객명, 위험·가치 지표,
SHAP 원본이 현재 DB의 저장 근거와 정확히 같은지 대조한 후에만 메모리에서 A4 PDF를 만든다. PDF도
DB에 저장하지 않는다. 인증은 서버의 `GEMINI_API_KEY` 또는 Vertex AI 환경변수를 사용하며 비밀은
프론트엔드에 노출하지 않는다. PDF 한글 폰트는 `RM_REPORT_FONT_PATH` 또는 OS fallback을 사용한다.

UI와 PDF의 “주요 SHAP Value (상위 10개)” 영역은 저장된 원본 행을 모두 표시하며 운영 적격 고객은
통상 10개를 표시한다. 서술에서 그룹화를 했더라도 행을 접거나 대표 피처만 보이게 하지 않는다.

## 9. RM 상담 가이드

| 신호 그룹 | 먼저 확인할 질문 | 허용되는 초기 접촉 방향 |
|---|---|---|
| 입출금 활동 | 매출·결제계좌 변경, 계절성, 타행 이체가 있었는가 | 주거래계좌·자금결제 구조 점검 |
| 자동이체 | 급여·공과금·매입대금 자동이체에 변화가 있었는가 | CMS·자동이체 현황 확인 |
| 채널 이용 | 이용 불편, 담당자 변경, 채널 전환이 있었는가 | 디지털 채널 지원·온보딩 검토 |
| 카드 관계 | 법인카드 사용처·한도·비용집행이 바뀌었는가 | 카드 이용조건·비용관리 상담 |
| 수신 관계 | 유동성 이동, 만기, 운전자금 사용이 있었는가 | 예치 구조·단기자금 운용 상담 |
| 여신 관계 | 상환, 만기, 신규 수요, 타행 대환이 있었는가 | 만기·한도·운전자금 수요 점검 |
| 관계 폭·구성 | 어떤 상품·채널이 먼저 변했는가 | 관계 회복 우선순위 설정 |

여신 또는 수신 잔액의 변화는 정상 상환·만기·사업 운영일 수 있다. SHAP 방향만으로 고객의 실제
원인이나 이탈을 단정하지 말고, 상담에서 사실을 확인한 뒤 고객 필요가 확인될 때만 적절한 제안을
검토한다.

## 10. 검수와 구현 흐름

생성 전·후와 배포 전에는 아래를 확인한다.

- 저장 SHAP이 56개 피처 집합, 유한 `impact`, 연속 rank, 1~10개 계약을 만족하는가.
- `localShapTop10`의 원본 행과 rank가 API, UI, PDF에 모두 보존되는가.
- `featureValue: null` 피처에 대해 프롬프트가 값·금액·변화율 추정을 금지하는가. 생성 결과의 숫자와
  사실은 RM 또는 원천 근거로 추가 확인했는가.
- `top10AbsShare`를 전체 모델 기여도 또는 확률 변화로 표현하지 않았는가.
- 적용 가능한 피처 방향 대조와 정량 SHAP-확률·위험 연결 검사를 통과했는가. 일반 인과 표현은 RM이
  추가 검토했는가.
- 표준 3개 한계문구가 포함됐고, 실제 해지·부도·휴면 확률로 오해할 표현이 없는가.
- PDF 요청이 현재 DB 근거와 불일치하면 거부되며, AI 보고서와 PDF가 DB에 저장되지 않는가.

운영 흐름은 다음과 같다.

1. 서비스 준비 단계에서 모델 점수, CLV, 저장 Local SHAP Top 10을 계산·저장한다.
2. 보고서 생성 API가 적격 고객의 저장 컨텍스트만 조회한다.
3. 규칙 엔진이 SHAP을 검증하고 피처별·그룹별 설명 맥락을 추가한다.
4. Gemini가 고정 JSON 서술을 생성하고, 서버가 스키마와 제한된 안전 후처리를 적용한다.
5. API가 저장된 정량 지표와 원본 SHAP 행을 다시 결합해 화면에 전달한다.
6. PDF 요청 시 서버가 근거를 재대조한 뒤, 모든 SHAP 행과 한계문구를 포함한 A4 PDF를 메모리에서 렌더링한다.

파인튜닝을 한다면 입력 근거 충실도, 피처 방향 일치, 근거 없는 숫자·원인 생성 여부, 표준 한계문구,
그룹화에 따른 반복 억제를 평가한다. 위험 모델 학습과 Gemini 문체 학습은 별개의 과정이다.
