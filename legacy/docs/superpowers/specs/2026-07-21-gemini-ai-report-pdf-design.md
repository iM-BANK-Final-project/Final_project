# Gemini AI 전략 보고서 및 PDF 다운로드 설계

## 목적

AI 리포트 페이지의 `전략 보고서 생성` 버튼을 실제 Gemini 생성 API와 연결한다. 선택 고객의 검증된 모델·CLV·SHAP·추천 데이터를 근거로 구조화된 한국어 전략 보고서를 화면에 보여주고, 사용자가 별도의 `PDF 다운로드` 버튼으로 동일한 내용을 내려받게 한다.

생성된 보고서와 PDF는 DB에 저장하지 않는다. Gemini API 키는 백엔드에서만 사용한다.

## 사용자 흐름

1. 사용자가 AI 리포트 페이지에서 고객을 선택한다.
2. `전략 보고서 생성`을 누른다.
3. 생성 중에는 버튼을 비활성화하고 `보고서 생성 중...`을 표시한다.
4. 생성에 성공하면 사전 저장 리포트 아래에 AI 보고서를 고정 섹션 형태로 표시한다.
5. 생성 완료 후에만 `PDF 다운로드` 버튼을 보여준다.
6. PDF 버튼을 누르면 현재 화면에 표시된 보고서와 동일한 내용의 PDF를 백엔드가 생성해 다운로드한다.
7. 고객을 변경하면 이전 AI 보고서 상태를 초기화한다.

## 아키텍처

### 보고서 생성 API

`POST /api/reports/{corporate_id}/generate`

- 요청 query: `as_of_month`
- 백엔드는 `ServiceRepository.report()`에서 선택 고객의 저장 리포트 컨텍스트를 조회한다.
- 컨텍스트에는 고객 식별정보, 기준월, 위험확률, CLV_Risk, PotentialLoss, D/A/C/K 변화율, SHAP Top 10, 기존 접촉 추천을 포함한다.
- 백엔드는 `gemini_service.py`의 전략 보고서 전용 함수를 호출한다.
- Gemini 응답을 구조화된 스키마로 검증한 뒤 JSON으로 반환한다.
- 고객이 없으면 404, Gemini 설정이 없거나 외부 호출이 실패하면 비밀정보를 노출하지 않는 502 응답을 반환한다.

### PDF 생성 API

`POST /api/reports/{corporate_id}/pdf`

- 요청 body는 화면에 표시된 구조화 보고서 JSON이다.
- 백엔드는 body를 보고서 스키마로 다시 검증한다.
- 백엔드는 고객을 DB에서 다시 조회해 고객ID, 기준월, 정량 지표와 요청 body의 메타데이터가 일치하는지 확인한다.
- PDF는 메모리에서 생성하고 파일시스템이나 DB에 저장하지 않는다.
- 응답은 `application/pdf` 바이너리와 한국어 파일명을 위한 적절한 `Content-Disposition` 헤더를 제공한다.

## 구조화 보고서 스키마

백엔드 응답과 PDF 요청은 동일한 보고서 envelope을 사용한다. 필수 필드:

- `corporateId`: 법인ID
- `customerName`: 화면 표시 고객명
- `asOfMonth`: 기준월
- `generatedAt`: ISO 8601 생성 시각
- `metrics`: 백엔드가 결합한 위험확률, CLV_Risk, PotentialLoss
- `shapFactors`: 백엔드가 결합한 순위 1~10의 SHAP 항목
- `riskSummary`: 종합 위험 요약
- `valueAssessment`: CLV_Risk·PotentialLoss 해석
- `weakeningDrivers`: 주요 약화 원인
- `contactStrategy`: RM 접촉 전략
- `recommendedActions`: 실행 권고사항 배열
- `caveats`: 분석 유의사항 배열

정량 지표와 SHAP Top 10은 Gemini가 재생성하지 않고 백엔드의 저장 컨텍스트를 envelope에 그대로 결합한다. Gemini가 반환하는 부분은 6개 서술 섹션뿐이다. 이로써 수치 변조와 환각 가능성을 줄인다.

## Gemini 프롬프트 계약

전략 보고서 프롬프트는 다음을 강제한다.

- 한국어 기업금융 RM 업무 언어를 사용한다.
- `risk_probability`를 향후 6개월 `Y_INTERVENE_M12_v2` 지속거래약화 발생 가능성으로만 설명한다.
- 실제 해지, 부도, 확정 휴면 확률로 표현하지 않는다.
- CLV_Risk와 PotentialLoss를 확정 손실액으로 단정하지 않는다.
- SHAP은 인과관계가 아닌 모델 예측 기여도로 설명한다.
- 입력 컨텍스트에 없는 재무정보, 사건, 상품 보유 사실을 만들지 않는다.
- 추천은 RM의 검토가 필요한 초기 접촉 전략으로 표현한다.
- 응답은 지정된 JSON 스키마를 준수한다.

## UI 상태

- 생성 전: 기존 저장 리포트와 SHAP Top 10, 생성 버튼 표시
- 생성 중: 생성 버튼 비활성화, 로딩 문구 표시
- 생성 완료: 6개 고정 섹션과 PDF 다운로드 버튼 표시
- 생성 실패: 기존 저장 리포트는 유지하고 생성 오류와 재시도 버튼 표시
- PDF 생성 중: 다운로드 버튼만 비활성화
- PDF 실패: AI 보고서는 유지하고 PDF 오류만 표시
- 고객 변경: AI 보고서, 생성 오류, PDF 오류 상태 초기화

## PDF 계약

- A4 세로 레이아웃
- 한국어 폰트 포함
- 표지/헤더에 기업명, 법인ID, 기준월 표시
- 위험확률, CLV_Risk, PotentialLoss 요약 표시
- 구조화 보고서 6개 섹션 표시
- SHAP Top 10을 순위, feature, SHAP value 표로 표시
- 생성 시각, 현재 모델 정의, SHAP/CLV 유의사항 표시
- 페이지 번호와 안정적인 페이지 나눔 적용

## 오류 및 보안

- Gemini 자격 증명은 `GEMINI_API_KEY` 또는 기존 `gemini_service.py`가 지원하는 Vertex AI 환경변수를 사용한다.
- 환경변수 값과 외부 API 원문 오류를 클라이언스에 노출하지 않는다.
- CORS에는 필요한 POST 메서드를 추가한다.
- PDF 요청의 구조, 문자열 길이, 배열 항목 수를 제한한다.
- PDF 요청의 고객 정보와 DB 컨텍스트가 다르면 400을 반환한다.
- 동시 중복 생성은 프론트엔드 상태로 우선 방지한다.

## 테스트 전략

### 백엔드

- 컨텍스트가 위험확률, CLV_Risk, PotentialLoss, D/A/C/K, SHAP Top 10, 추천을 포함하는지 검증
- Gemini 구조화 응답 성공, 필수 섹션 누락, 빈 응답, 외부 API 실패 검증
- 생성 API의 200, 404, 502 응답 검증
- PDF API의 정상 PDF 시그니처, content type, 파일명 검증
- PDF 요청 메타데이터 불일치와 스키마 오류 검증
- 생성된 PDF의 한국어 텍스트 추출 및 페이지 렌더링 검증

### 프론트엔드

- 생성 버튼이 POST를 호출하고 로딩 상태를 표시하는지 검증
- 성공 시 6개 섹션과 PDF 버튼이 표시되는지 검증
- 실패 시 기존 리포트가 유지되고 재시도가 가능한지 검증
- PDF 다운로드가 blob, 파일명, URL 정리를 올바르게 처리하는지 검증
- 고객 변경 시 이전 AI 보고서를 초기화하는지 검증

### 실제 호출

자동화 테스트에서는 Gemini를 대체하여 반복 실행과 비용 발생을 막는다. 모든 테스트 통과 후 현재 `.env`로 선택 고객 1건을 수동 통합 검증한다. 통합 검증에서는 비밀 값을 로그에 출력하지 않는다.

## 완료 기준

- 선택 고객의 Gemini AI 전략 보고서가 6개 고정 섹션으로 UI에 표시된다.
- 생성 후 별도의 PDF 버튼으로 동일한 보고서를 다운로드할 수 있다.
- PDF에 한국어, 정량 지표, SHAP Top 10, 생성 시각, 유의사항이 정상 표시된다.
- 적격 3,341개 서비스 모집단 밖의 고객은 보고서 생성 대상이 될 수 없다.
- API 키가 프론트엔드 번들이나 API 응답에 포함되지 않는다.
- 백엔드·프론트엔드 자동화 테스트와 프론트엔드 프로덕션 빌드가 통과한다.
