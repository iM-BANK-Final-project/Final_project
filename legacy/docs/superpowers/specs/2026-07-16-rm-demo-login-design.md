# RM Copilot 로컬 데모 로그인 설계

## 1. 목적

`src/frontend/rm-insight-copilot`의 기존 React 서비스 앞에 로컬 시연용 로그인 화면을 추가한다. 사용자가 정해진 데모 계정으로 로그인하면 현재 iM Bank 스플래시 애니메이션을 거쳐 기존 RM 서비스 화면으로 진입한다.

이번 작업은 실제 사용자 인증 구축이 아니다. 현재 FastAPI 조회 API, 서비스 페이지, 페이지 상태 기반 전환 구조는 변경하지 않는다.

## 2. 범위

### 포함

- 첫 진입 시 로그인 화면 표시
- 제공 이미지와 기존 `Login.js`를 참고한 좌측 브랜드 이미지·우측 로그인 폼 레이아웃
- 데모 계정 `test / 1234` 검증
- 비밀번호 표시·숨김과 Enter 제출
- 로그인 실패 시 인라인 오류 안내
- 로그인 성공 후 기존 iM Bank 스플래시 애니메이션 표시
- 스플래시 종료 후 현재 Overview 및 서비스 화면 표시
- `sessionStorage` 기반 탭 단위 로그인 유지
- 데스크톱과 좁은 화면을 지원하는 반응형 레이아웃
- 로그인 및 화면 전환의 컴포넌트 테스트

### 제외

- FastAPI 로그인 API와 사용자 테이블
- 비밀번호 해시, 세션 쿠키, JWT, 권한 관리
- 회원가입, 아이디 찾기, 비밀번호 찾기
- 로그아웃 기능
- 기존 분석·추천·보고서 데이터 흐름 변경

## 3. 사용자 흐름

```text
새 브라우저 탭에서 접속
  → 로그인 화면
  → test / 1234 입력
  → 로그인 성공
  → iM Bank 스플래시 애니메이션
  → 기존 RM 서비스 Overview
```

잘못된 계정을 제출하면 로그인 화면을 유지하고 `아이디 또는 비밀번호가 일치하지 않습니다.`를 표시한다. 입력값이 비어 있어도 같은 방식으로 로그인에 실패한다.

로그인 성공 시 `sessionStorage`에 인증 표시를 저장한다. 같은 탭에서 새로고침하면 스플래시를 반복하지 않고 서비스 화면으로 바로 복원한다. 탭을 닫아 브라우저 세션이 끝난 뒤 다시 접속하면 로그인 화면부터 시작한다.

## 4. 화면 설계

### 데스크톱

화면 전체를 두 영역으로 나눈다.

- 왼쪽: 제공된 `Gemini_Generated_Image_1s58u01s58u01s58.png`를 표시하는 브랜드 패널
- 오른쪽: `로그인` 제목, 아이디 입력, 비밀번호 입력, 비밀번호 보기 버튼, 로그인 버튼

브랜드 이미지는 React 앱의 정적 자산으로 복사해 외부 파일 경로에 의존하지 않게 한다. 이미지 비율을 유지하고 패널 안에서 잘리지 않도록 `object-fit: contain`을 사용한다.

기존 `Login.js`의 입력 구성과 비밀번호 보기 동작은 참고하되, 1440×1024 절대 좌표와 인라인 스타일은 가져오지 않는다. 현재 앱의 CSS 파일에서 재사용 가능한 클래스 기반 반응형 스타일로 구현한다.

### 좁은 화면

두 영역을 세로로 배치한다. 브랜드 이미지는 위, 로그인 폼은 아래에 표시하며 입력 요소가 화면 너비를 넘지 않도록 한다.

### 접근성

- 모든 입력에 연결된 label을 제공한다.
- 로그인 폼의 submit으로 버튼 클릭과 Enter 입력을 동일하게 처리한다.
- 비밀번호 보기 버튼은 현재 상태를 설명하는 `aria-label`을 갖는다.
- 로그인 오류는 `role="alert"`로 즉시 인식할 수 있게 한다.
- 키보드만으로 모든 로그인 동작을 수행할 수 있게 한다.

## 5. 컴포넌트와 상태 경계

### `LoginPage`

로그인 화면의 표시와 폼 상태만 담당한다.

- `username`, `password`, `showPassword`, `error`를 관리한다.
- submit 시 `onLogin(username, password)`를 호출한다.
- 인증 성공 여부는 상위 `App`이 결정한다.

### `App`

앱의 진입 단계를 관리한다.

```text
login → splash → service
```

- 초기 `sessionStorage` 값이 없으면 `login` 단계다.
- `test / 1234`가 아니면 `login` 단계를 유지하고 실패 결과를 반환한다.
- 올바른 계정이면 인증 표시를 저장하고 `splash` 단계로 전환한다.
- 기존 2.7초 타이머가 끝나면 `service` 단계로 전환한다.
- 이미 인증된 세션이면 초기 단계를 `service`로 정해 새로고침 시 스플래시를 재생하지 않는다.

기존 `activePage`, `selectedCustomerId`, `TopNav`, 각 서비스 페이지의 props 전달은 `service` 단계 안에서 그대로 유지한다.

## 6. 인증 데이터 취급

데모 계정은 프론트엔드 번들에 포함되므로 누구나 확인할 수 있다. 따라서 코드에 로컬 시연 전용임을 명시하고 실제 인증 또는 보안 경계로 표현하지 않는다.

`sessionStorage`에는 아이디나 비밀번호를 저장하지 않고, 현재 탭이 인증되었다는 불리언 성격의 표시만 저장한다. 저장 키는 앱 전용 이름을 사용한다.

## 7. 오류와 예외 처리

- 잘못된 계정: 폼 아래에 오류 표시, 서비스 진입 차단
- 로그인 재시도: submit 시 이전 오류를 갱신
- 저장소 접근 불가: 로그인 성공 후 현재 React 상태로는 서비스 진입을 허용하되 새로고침 유지 기능만 포기
- 스플래시 컴포넌트: 기존 애니메이션과 타이머를 그대로 사용
- 백엔드 API 오류: 로그인과 분리하며 기존 각 페이지의 오류 상태 처리 유지

## 8. 테스트 계약

컴포넌트 테스트는 다음 동작을 검증한다.

1. 인증 표시가 없으면 브랜드 이미지와 로그인 폼이 먼저 보인다.
2. 잘못된 아이디 또는 비밀번호로 submit하면 오류가 표시되고 스플래시·서비스는 보이지 않는다.
3. `test / 1234`로 submit하면 스플래시가 보인다.
4. 타이머가 끝나면 기존 서비스 내비게이션과 Overview가 보인다.
5. 인증 표시가 있는 상태로 렌더링하면 로그인·스플래시 없이 서비스가 바로 보인다.
6. 비밀번호 보기 버튼이 input type을 `password`와 `text` 사이에서 전환한다.

테스트는 실제 FastAPI 호출 없이 앱 진입 흐름만 검증한다. 기존 서비스 페이지 테스트와 API 클라이언트 테스트는 변경 없이 계속 통과해야 한다.

## 9. 완료 조건

- 새 탭 첫 진입에서 로그인 화면이 표시된다.
- `test / 1234` 외의 계정으로는 서비스에 진입할 수 없다.
- 로그인 성공 후 iM Bank 애니메이션을 거쳐 기존 서비스가 표시된다.
- 새 로그인 화면이 기존 서비스 화면과 API 데이터 연결을 변경하지 않는다.
- 데스크톱과 좁은 화면 모두에서 이미지와 폼이 잘리거나 겹치지 않는다.
- 관련 신규 테스트, 전체 프론트 테스트, 프로덕션 빌드가 통과한다.
